NASA Software Defect Datasets

Outline

"When a measure becomes a target, it ceases to be a good measure."

Goodhart's Law

The NASA Software Defect Datasets is a collections of overlapping datasets dealing with predicting software failure from static code metrics. The datasets are overlapping in the sense that many cases appear in multiple datasets, but since all of the individual datasets suffer from various data issues there is no authoritative/reference dataset.

The aim of this assignment is to take one of these datasets, clean it, and construct a classifier using the cleaned dataset.

The cleaning process itself is not complicated but the complexity arises from the fact that you will need to read and parse a paper that describes the required process rather than following my usual instructions (which are obviously always correct and always very clear ðŸ™ƒ).

Details

To start you need to read the paper Data Quality: Some Comments on the NASA Software Defect Data Sets [1] which describes the cleaning process used to standardise the multiple datasets. We (you) will apply this process to one of the NASA Software Defect datasets and then build a classifier.

The features are based on the work of McCabe and Halstead. You don't need to read those articles in order to do this assignment, but I found the development of static code metrics interesting as a "want to be programmer".

Step 1: Import

Of the datasets listed in (see Table 1 of [1]) we will work on the PC2 dataset which comes from the Promise Repository, so is listed as "PC2 Promise", with 5589 cases and 37 features (actually 36 features + 1 target), in the paper (see Table 1 of [1]).

Step 2: Data Clean

Clean the PC2 dataset using the process described in [1].

However, it should be noted that this paper, (if I'm correct) contains errors — ironically for a paper covering errors in datasets. So use its results as a guide but don't be too surprised if you get different results.

Step 3+: Model Building

The classification step should be relatively straightforward, but a few observations are worth noting:

  • The number of cases is small (especially after cleaning) so you will need to relay on cross validation scoring rather than train/validation/test split. In fact, I would probably not do a train/test split at all.

  • The dataset is imbalanced and in addition to the low number of cases we need to be careful whenever preforming a test/train or CV split and will need to apply imbalanced techniques (think SMOTE).

  • Accuracy is a uninformative measure of success for a defect detector, because two detectors with the similar accuracy can have widely varying True Positive and False Positive rates. Hence we will use PR AUC as our metric. A good article on the PR AUC metric (and more) is The Precision-Recall Plot Is More Informative than the ROC Plot When Evaluating Binary Classifiers on Imbalanced Datasets

  • Feature importance is relevant, so you should be able to make some comments on which features are important, and their relative importance etc.

Upload the following notebooks (and additional python scripts if used):

  • 01-Clean.ipynb

  • 02-Model.ipynb

  • Performs feature engineering, model selection and optimisation.
  • Generates a suitable feature importance summary.
